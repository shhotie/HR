Can you describe a challenging situation in your project and how you resolved it?

Yes, one challenging situation I faced was during the development of the report cloning feature in the SmartReg project. The requirement was to allow users to duplicate an existing regulatory report—along with all its sections, rules, and metadata—so they could reuse it for the next period.

The challenge came when cloning reports that had complex nested relationships, like validation rules linked to specific sections, and user permissions tied to those sections. Initially, our clone logic was copying only the top-level report structure but missed these linked entities. This led to broken references and missing data in the cloned reports, which affected user experience.

To resolve this:

  • I analyzed the entity relationships deeply and updated the clone logic to recursively copy all dependent entities, preserving relationships like foreign keys and ownership metadata.
  • I added transactional handling using @Transactional to ensure either the full report was cloned successfully or rolled back in case of an error.
  • We also implemented proper exception handling and user messages when a clone failed due to business rules like duplicate periods.

After the fix, the cloning feature worked smoothly, reduced manual effort for users, and also improved trust in the system.

Can you explain the architecture of your recent project?

Yes, I’d be happy to explain the architecture of my recent project, SmartReg, which is a cloud-based regulatory compliance platform developed for financial institutions.

High-Level Architecture:

Architecture Style:
We used a microservices-based architecture built using Spring Boot. Each module (such as report generation, approval workflow, rule validation, etc.) was handled by a separate microservice.

Backend:

  • Developed using Java 8, Spring Boot, and Spring MVC.
  • Each microservice exposed RESTful APIs for communication.
  • Used Spring Security with JWT tokens to secure the endpoints.

Frontend:

  • Built using Angular (version 12).
  • Consumed REST APIs exposed by the backend.
  • Handled dynamic form rendering based on report type and user permissions.

Database:

  • PostgreSQL used for data storage.
  • Each service managed its own schema following Domain-Driven Design (DDD).
  • Audit logs and workflow data stored in dedicated tables.

Workflow Engine:

  • Implemented a custom 3-step approval workflow (Maker → Checker → Reviewer).
  • Configurable and role-based.

DevOps and Deployment:

  • Docker used to containerize services, with Docker Compose in non-prod environments.
  • Jenkins handled CI/CD: builds, tests, and deployments.
  • Services deployed in a cloud-based environment, monitored via Spring Boot Actuator and logs.

Authentication/Authorization:

  • User roles managed via Spring Security.
  • JWT tokens issued upon login for secure access control.

Inter-Service Communication:

  • Mainly RESTful.
  • Used asynchronous event-driven handling in some scenarios.

Why This Architecture?
This architecture was chosen for its scalability, modularity, and clear separation of concerns. Since SmartReg serves multiple financial clients with configurable rules and validations, microservices offered the flexibility and isolation required for continuous delivery.

What challenges did you face while working with microservices?

While working with microservices in the SmartReg project, I encountered several challenges, including:

1. Service Communication & Dependency Management:

One of the main challenges was handling inter-service communication efficiently. For example, when the reporting service needed data from the validation and workflow services, we had to ensure APIs were always available and responsive. To address this:

  • We designed clear contracts and fallback logic.
  • Implemented timeout settings and proper exception handling to avoid cascading failures.

2. Data Consistency:

Each service had its own database schema (as per Domain-Driven Design). This led to eventual consistency challenges, especially in workflows where multiple services had to act in sequence (e.g., validation → approval → report submission). To solve this:

  • We used transaction logs and ensured updates were processed in the correct order.
  • In critical cases, we built compensating transactions.

3. Debugging and Tracing Across Services:

When an issue occurred, it wasn’t always clear which service failed. So we:

  • Enabled centralized logging using log aggregators.
  • Used unique request IDs passed between services for better tracing.

4. Deployment and Configuration:

Managing multiple services meant keeping their configurations consistent across environments. To manage this:

  • We used Docker and Docker Compose for local development.
  • Ensured centralized config management using environment variables and profiles.

5. Version Compatibility:

As multiple services were developed and deployed independently, there were instances where one service’s API change broke another. We mitigated this by:

  • Following strict API versioning.
  • Using Swagger for documentation and automated contract validation during CI builds.

In short, microservices gave us flexibility and scalability, but required strong discipline in design, logging, and deployment practices — all of which I actively contributed to during the SmartReg project.

How do you ensure the security of your APIs?

Security Practices in the SmartReg Project

1. Authentication and Authorization:

We used Spring Security along with JWT (JSON Web Tokens).

After a successful login, a JWT token is issued to the user, and this token must be included in the headers for all secured API requests. Role-based access control (RBAC) was applied to ensure that users could only access resources allowed by their roles (e.g., Maker, Checker, Reviewer).

2. Input Validation and Sanitization:

Validations were performed both at the controller and service layer using annotations like @Valid, @NotNull, etc.

We prevented injection attacks (like SQL injection or XSS) by using parameterized queries in Hibernate and avoiding direct user inputs in queries.

3. HTTPS and Secure Headers:

All APIs were deployed behind HTTPS to encrypt data in transit. We also added security headers (like X-Content-Type-Options, X-Frame-Options, etc.) in responses to mitigate common web vulnerabilities.

4. Rate Limiting and Throttling (planned/future-ready):

To avoid brute-force attacks and DoS, we had designs in place for implementing rate limiting using tools like API Gateway or filters.

5. Audit Logging and Monitoring:

User actions and API access were logged in audit tables. Logs were monitored for unauthorized access patterns or anomalies using log aggregators.

6. Exception Handling and Masking Sensitive Data:

Centralized exception handling using @ControllerAdvice was implemented to ensure stack traces or sensitive errors are not exposed to clients. We returned only generic messages in error responses while logging the actual error details internally.

In short, API security is a layered approach — we ensured it through proper authentication, authorization, data validation, secure communication, and ongoing monitoring.

What is your approach to logging, monitoring, or debugging issues in production?

Logging and Monitoring Practices in the SmartReg Project

1. Structured Logging:

We use SLF4J with Logback for consistent and readable logging.

Logs are structured with unique request IDs, timestamps, log levels (INFO, DEBUG, ERROR), and context (like user ID or module name).

For sensitive data, we ensure masking before logging.

2. Log Levels and Categories:

I configure appropriate log levels — keeping DEBUG for local/dev, and INFO/ERROR for production.

In case of exceptions, we always log the root cause (e.getMessage() and stack trace), which helps during debugging.

3. Centralized Logging:

All logs are collected centrally using log aggregators (like ELK stack — Elasticsearch, Logstash, Kibana) or Fluentd with a dashboard.

This allows the team to search logs across services, filter by user ID, and identify issues quickly.

4. Monitoring Tools:

We use Spring Boot Actuator for health checks and endpoint monitoring.

Dockerized services have built-in health check scripts, which are also monitored by the deployment environment (like Jenkins/Kubernetes).

5. Alerting & Notifications:

For critical errors or service downtimes, we configure alerts via email or Slack using monitoring tools.

This ensures we're immediately aware of any production issues.

6. Debugging in Production:

We never debug directly in production but use:

  • Detailed logs
  • Audit trails
  • Test case recreation in staging environments

If needed, we simulate production data in lower environments to reproduce and fix issues safely.

Example:

Once in SmartReg, we had a report generation issue that wasn't reproducible in dev. By using the trace ID from logs, we correlated API calls and discovered that a null value from a dependent service caused a failure. The issue was resolved and we added a null check with a fallback value in the service layer.
